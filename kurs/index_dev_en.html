01
Architecture — Files Instead of a Database

No SQLite, no Redis, no vector DB. Context Memory stores everything as JSON and Markdown in the file system. This makes it portable, inspectable, and trivially debuggable.

~/.context-memory/ ├── mein-projekt/ │ ├── tree.json ← Baumstruktur mit Parent/Child-Refs │ ├── index.json ← Flacher Such-Index: ID→{title,type,tags,relevance,confidence,temporal} │ ├── relations.json ← Kanten-Liste: [{from,to,relation,note,created}] │ ├── history.json ← Audit-Log, max 500 Einträge, FIFO │ └── nodes/ │ ├── arch-001.md ← Markdown mit YAML-Header (Typ, Relevanz, Confidence, Temporal) │ ├── dec-001.md │ └── les-001.md └── anderes-projekt/ └── ... ← Komplett isolierter Workspace
Warum JSON statt SQLite?
Weil JSON schneller ist
Weil Claude keinen Zugriff auf Datenbanken hat
Weil es portabel, inspizierbar und ohne externe Dependencies funktioniert

Warum braucht generate_id() einen optionalen in-memory Index?
Weil bei Batch-Inserts der Disk-Index noch nicht aktualisiert ist und sonst alle IDs gleich wären
Weil der Index zu groß für den Arbeitsspeicher sein könnte
Weil es threadsicher sein muss
04
Agentic Search — 3 Stages Without a Vector DB

Keine Embeddings, kein FAISS, kein Cosine Similarity. Stattdessen: gewichtete Tag/Title-Matches, dann Fulltext mit Phrase-Bonus, dann Relations-Anreicherung.

# Stufe 1: Quick Search (cm_search.py) score = 0.0 tag_matches = query_words & tags # × 3.0 Gewicht title_matches = query_words & title_words # × 2.0 Gewicht substring_matches ... # × 1.0 / 0.5 score *= (1 + relevance_boost * 0.2) # critical → +30% # Stufe 2: Deep Search — nur wenn Quick < limit for qw in query_words: count = content_lower.count(qw) score += min(count, 5) * 0.5 # Diminishing returns # Stufe 3: Relations (mit --with-relations) rels = get_node_relations(ws, result_id) # Zeigt: wovon hängt der Treffer ab, was behebt er, etc.

05
Knowledge Graph — Adjacency List in JSON

Der Graph ist eine einfache Kanten-Liste in relations.json. 22 typisierte Relationstypen, jeder mit automatischer Umkehrung. BFS für Pfadfindung, N-Hop-Traversal für Kontextabruf.

# relations.json — einfache Edge-Liste { "edges": [ {"from": "arch-001", "to": "dec-001", "relation": "depends_on", "note": "API braucht DB"}, ... ] } # BFS-Pfadfindung in cm_relate.py def find_path(ws, from_id, to_id, max_depth=6): adj = {} # Adjazenzliste aus edges bauen for edge in edges: adj[edge["from"]].append((edge["to"], edge["relation"])) # Auch reverse traversieren! adj[edge["to"]].append((edge["from"], f"~{rel}")) # Standard-BFS mit visited-Set queue = deque([(from_id, [(from_id, None)])])

Warum werden Emojis aus den Embedding-Chunks entfernt?
Weil Emojis zu viele Tokens verbrauchen
Weil Embedding-Modelle sauberen Text ohne Formatierungs-Artefakte brauchen
Weil JSON keine Emojis unterstützt
07
The Big Picture — and What is Missing

Context Memory ist ein SkillEine Markdown-Datei (SKILL.md) die Claude Anweisungen gibt, plus unterstützende Scripts. Claude liest die SKILL.md und führt die Scripts im Hintergrund aus., kein eigenständiges Tool. Claude liest die SKILL.md, versteht den Workflow und ruft die Scripts über bash auf. Der Nutzer sieht nur natürliche Sprache.

Was gut gelöst ist: Dateisystem-Portabilität, Duplikat-Erkennung, in-memory Batch-IDs, saubere Chunk-Bereinigung, 22 Relationstypen mit BFS.

Was man erweitern könnte: Semantic Search via Embeddings (braucht API-Key), Real-time Collaboration (braucht Server), Automatisches Kuratieren bei jedem Gespräch (braucht Hook-Integration), Graph-Visualisierung als interaktives HTML.
